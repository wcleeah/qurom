## Why This Phase Exists

Phase 07 gets the TUI functional, but its layout is deliberately provisional. If Phase 08 wires keybindings, focus order, and footer hints against a temporary layout, we will either rework those bindings immediately afterward or cement a weaker screen structure into the product. This phase stabilizes the screen hierarchy before the keyboard contract is added.

The key design decision is to stop treating all four agents as equal-width peers. In this product, the drafter is the primary artifact producer and the auditors are review/support lanes. The UI should reflect that.

## Implementation Details

### Running screen hierarchy

Target composition:

```tsx
<box flexDirection="column" flexGrow={1}>
  <Dashboard />
  <box flexDirection={wide ? "row" : "column"} flexGrow={1}>
    <DrafterPane />
    <AuditorRail />
  </box>
  <Footer />
  <SystemStatusSurface />
</box>
```

Notes:

- `DrafterPane` can initially be a light wrapper around `AgentPanel` with a main-pane variant; no need to invent a separate data model.
- `AuditorRail` is a stacked column of compact auditor cards on wide terminals.
- On narrow terminals the layout collapses to a vertical stack, still with the drafter first.
- Avoid a full metadata sidebar until there is enough data to justify it; a compact header plus transient status surface is enough for this phase.

### Prompt screen redesign

Use `reference/opencode/.../routes/home.tsx` as the composition reference:

- one dominant centered composer region
- compact mode switch near the composer rather than a separate form block above it
- document mode should still show path, size, and first-line preview, but as part of the main compose surface instead of a detached status box

### Summary screen redesign

Shift from `result` / `next` boxes to:

- prominent outcome badge/title
- compact run facts row (`round`, approvals, unresolved findings)
- artifact block (`output path`, `trace id`, error if any)
- action list or action selector with stronger wording and no stub-like copy

### Theme expansion

Extend `src/tui/theme.ts` to support lighter chrome and clearer contrast:

- `background`
- `backgroundPanel`
- `backgroundElement`
- `borderSubtle`
- `borderActive`
- `text`
- `textMuted`
- `accent`
- `success`
- `warning`
- `error`

Keep the existing role colors, but stop using border color as the only source of hierarchy.

### Footer and status surfaces

- `Footer` should only show controls that are already real in Phase 07.5.
- Add an optional `SystemStatusSurface` or toast-style component for transient warnings/errors so operational status does not have to live inside the main content panes.
